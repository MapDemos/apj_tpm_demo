# L0 会話レイヤ 設計メモ（2026-07-10）

> ステータス：**設計合意・実装未着手**。本メモは相談で固めた合意の記録であり、コードは
> まだ変更していない。実装時はここを一次ソースとする。

---

## 0. 目的

現在の **L1-1（`confirmInput`＝一発の復唱）** を、独立した **会話レイヤ L0** に昇格する。

- L0 は「自分はジオネーター＝**場所を特定する**のが仕事」と自覚し、ユーザーと自然な会話をする。
- L0 は裏で**決定的な推論・計算パイプライン（現 L1-2 以降）をキック**する。
- **会話（L0）と 計算（エンジン）を棲み分ける**。会話 LLM はジオ判断・数値計算をしない。
- 次段階として、この会話部分を**音声化**するオプションを作る。

これは以前 agentic-loop（LLM 主導）を捨てて JS 主導にした流れと矛盾しない：**JS を計算の
オーケストレーターに据えたまま、その "前" に会話 LLM を乗せる**。LLM は対話と UX だけを司る。

---

## 1. 全体アーキテクチャ（3 層）

```
[L0] 会話エージェント（新設・ステートフル・LLM）
      ├ 使命理解・トーン・多ターン・音声
      ├ インテント・ルーター（発話が今どの操作かを振り分ける）
      └ 待ちナレーションの唯一の所有者
        ↑↓  ← 接ぎ目は既存 UICallbacks（後述）
[エンジン] JS オーケストレーター（既存・据え置き・決定的）
      ├ 現 L1-2 parse（text→QuerySchema, PROMPT_L1）
      ├ L2-1 カテゴリ / L2-2 関連性 / L3 目印
      └ proximity 解決・収集・評価・スコア・tier
        ↓
[外部] Mapbox（Search Box / Tilequery / Isochrone / Static / GL）＋ Turf
```

### 番号の扱い
- **L1-1（`confirmInput`）：コードは残し、機能はオフ**（削除しない）。必須機能ではなく、L0 の
  初手挨拶と重複するため停止する。
- **L1-2 以降：ロジック据え置き**。**リナンバリングはしない**（L1-1 を残すため番号は現状維持）。

## 2. 決定性の契約（この設計の生命線）

看板「**同じ質問 → 同じ答え**」を守るため、決定性の線を **2 段** に引く。

| 層 | 担当 | 決定性 |
|---|---|---|
| **振り分け（インテント・ルーティング）** | **L0** | 非決定でよい（誤ルートは会話で訂正できる） |
| **意味の確定（geo 解釈・距離・スコア・tier・確度）** | **エンジン** | **決定的（必ず）** |

厳守事項：

1. **L0 がエンジンに渡すのは「ユーザーの実際の言葉」（or 決定的な正規化）だけ。**
   LLM で言い換えた要約を渡すのは**禁止**（そこを通すと入力が非決定になり看板が崩れる）。
2. **確度ラベル（突出／団子など）の "判定" は JS が決定的に行う**。L0 は**言い換えるだけ**。
   ＝既存ルール「数値・判定は JS、言葉は LLM」そのもの（新しい例外ではない）。
3. L0 の**言い回し・雑談・断り**は非決定でよい（geo の答えではないため）。

## 3. 用語：処理ビュー と モード

- **処理ビュー**：エンジンの構造化 UI 一式の総称 ＝ **候補パネル・ボタン類・地図描画（ピン/bbox/
  到達圏）・デフォルトクエリ・進捗**。
- **詳細モード（処理ビュー ON・既定）**：全部見せる ＋ L0 も全部を解説（重複を許容する "全部見せ"）。
- **会話モード（処理ビュー OFF）**：パネル/ボタン/デフォルトクエリ/進捗を隠し、L0 は簡潔（会話・音声向け）。

> **既定は詳細モード**（JS 主導の推論を "見せる" のがこのデモの売り）。会話/音声を見せたい時にトグル。

### プロース（自然文）は全部 L0 が飲む
`〜が見つかりませんでした`（`condNotFound`）／`〜周辺に絞って探します`（overflow note）／除外・申し送り
（`_exclusionNote`）／エラー（`_friendlyError`・`not_a_query`）／結果サマリー／確度コメント。
→ 従来の定型文は通常モードでは出さず、**L0 が status を読んで言い換える**。定型文はデバッグで残す。

### 処理ビューとして残す（構造化・モードでトグル）
- **ボタン類**（選択肢・フィードバック・提案）
- **候補パネル**（結果一覧）
- **地図描画**（ピン/bbox/到達圏）※地図は "見るもの" なので L0 は代弁しない。地図自体は分けない。

### Static Images（サムネイル）は排他・処理ビュー優先
上位 5 件ティア色ピンの静的地図（`_buildStaticMapUrl`）は住所を常に 1 つに：
- 処理ビュー ON → 従来どおり候補パネル内。
- 処理ビュー OFF → **L0 の会話メッセージ内に差し込む**。
- 二重表示しない。生成ロジックは使い回し、貼る先だけモードで切替。

## 4. ① モード別・吹き出し別 発話対応表

| 出来事 | 判定/計算 | 詳細モード（既定） | 会話モード | デバッグ（運用者オーバーレイ） |
|---|---|---|---|---|
| **起動・初回** | L0 | greeting ＋ デフォルトクエリ chips | greeting のみ | 詳細 ＋ 運用情報 |
| **送信直後（受領）** | L0 | 即時 ack 一言 ＋ `・・・`。裏でエンジンを**楽観並列**キック | 同左 | 同左 |
| **処理中（待ち）** | L0 が唯一のナレーター | L0 の `・・・` ＋ 口語一言 | 同左 | **エンジン粒度ラベル**（場所特定/収集/評価）＋ ステップ停止パネル |
| **確認要求（どの入谷?）** | エンジンが「要確認」を判断／選択肢は決定的 | L0 が自然文で質問 ＋ **ボタンパネル** | **番号付き**選択肢を提示 → ユーザーが**番号を打つ**（ボタンなし）。番号→index は **JS が直接マップ（決定的）**、曖昧回答のみ L0 が拾う | ＋ 生の候補データ |
| **途中メッセージ（0件/絞り込み/除外/申し送り）** | エンジン（status） | **L0 が自然文**で伝える | 同左 | ＋ 定型文・生 status |
| **結果提示** | エンジン（決定的・tier/score） | 候補パネル（全件）＋ 地図ピン ＋ L0 サマリー | **番号付き**地図ピン（**tap 可**）＋ **L0 が上位 5 件を口頭** ＋ static 画像（排他）。番号を打って個別操作も可 | ＋ schema JSON・グリッド・TQ 内訳 |
| **確度コメント** | **JS が tier から確度ラベル（決定的）** | L0 が言い換え | 同左 | ＋ 生 tier・ラベル値 |
| **フィードバック（確定/絞り込み/探し直す）** | 選択肢＝JS | ボタン ＋ L0 一言 | L0 が口頭で促す → **番号 or キーワード**で回答（ボタンなし） | 同左 |
| **エラー** | エンジン | L0 が自然文 | 同左 | ＋ 生エラー（HTTP/打ち切り等） |
| **雑談・ミッション外** | **L0 ルーター** | L0 が人格のまま会話／off-mission タスクは断る（エンジン起動しない） | 同左 | 同左 |
| **キャンセル** | ユーザー（■ or 「待って」）→ 既存 `_cancelled`/`_runGen` | L0 が受領一言。入力は L0 が文脈保持（再入力不要） | 同左 | 同左 |

## 5. ② L0 ↔ エンジンの "口"（インターフェース）

**接ぎ目は新設しない。既存の `_buildUICallbacks`（index.js）がそのまま境界になる。**
今は「エンジン → UICallbacks → DOM 直行」。新設計は「エンジン → UICallbacks → **L0** →（DOM or 音声）」。
＝エンジンは今までどおり `ui.showChoices(...)` 等を呼ぶだけ。**変えるのはコールバックの "中身"**（DOM 直描画
から L0 経由へ）。これによりエンジンの意思決定ロジックは 1 行も触らずに済む。

### L0 → エンジン（駆動）
| 口 | 対応する既存経路 | 用途 |
|---|---|---|
| `startSearch(userText)` | `run(userText)` 相当 | 新規検索。実際の言葉を渡す。楽観並列可 |
| `refine(userText)` | `parseRefinement` / `_narrowWithin` / `_narrowByFloors` / feedback | 条件追加・絞り込み・探し直し |
| `cancel()` | `_cancelled=true` ＋ `_runGen` 無効化 | 中断 |
| `answerClarification(answer)` | 現 `showChoices`/`showHintInput` の resolve | 保留中の確認 await を解決 |

### エンジン → L0（既存 UICallbacks を再解釈）
| 既存コールバック | L0 での扱い |
|---|---|
| `showMessage` / `showSearching` / `thinking` | L0 が飲む（待ち・申し送りを口語化）。粒度ラベルはデバッグへ |
| `showChoices` / `showHintInput` | L0 が自然文で問う（詳細はボタン併用／会話はボタンなし） |
| `showFeedback` | 同上（確定/絞り込み/探し直す） |
| `showResults(full, partial, none, summary, …)` | 候補・tier・score を受け、L0 がサマリー＋確度を生成。static URL 差込先を切替 |
| `showDebug` / `showDebugReport` / `showRunStats` | デバッグモードのみ（従来どおり） |
| `draw*` / `fitToBBox` | 地図描画は従来どおり（L0 は関与しない） |

### 追加が要る口（新規）
- **status 購読**：進捗フェーズ（場所特定/収集/評価）を L0 が "読む" 経路（口に出すかはモード依存）。
- **score/確度 購読**：③ の確度ラベルを結果と一緒に受ける。
- **確認 await の中断対応**：`showChoices`/`showHintInput` の待ちを、回答だけでなく**キャンセルでも解ける**ように。

## 6. ③ 確度ラベルの JS 判定仕様

**既存の tier（full/partial/none）分布から JS が決定的にラベルを出す。L0 はラベル → 自然文の変換のみ。**
（別途のスコア差閾値は設けず、既に決定的な tier をそのまま確度の根拠にする。）

- 入力：上位候補の tier 分布（`_tier` / full・partial・none）。

| ラベル | 条件（tier ベース） | L0 の言い換え（例・毎回揺れてよい） |
|---|---|---|
| `decisive` | full が 1 件だけ（他は下位） | 「これは高い確率で当たりだと思います」 |
| `ambiguous` | full が複数（甲乙つけがたい） | 「確率順に並べましたが、正直つけがたいです」 |
| `tentative` | 最上位が partial | 「一番手はこれですが、確証まではという感じです」 |
| `none` | 該当なし（full/partial なし） | 「該当が見つかりませんでした」 |

- **確度の "主張" は毎回同じ（決定的）／言い回しだけ揺れる（許容）**。

## 7. 非同期分離とキャンセル（最大の実装難所）

- 今 `run()` は**ブロッキング**、`showChoices`/`showHintInput` は**同期 await**（回答が来るまで停止）。
- 新設計：**エンジン＝中断可能なバックグラウンドジョブ／L0＝常時 listening**。
- 土台は既存の `_cancelled`・`_runGen`・`_aborted(gen)`・`isCancelled()`。
- **肝は「確認 await を中断可能にする」**：待っている最中に
  - キャンセル（■/「待って」）
  - 別の発話（条件追加・訂正・雑談）
  が割り込んだ時に、await を安全に解いてレース（エンジン完了と L0 発話の衝突、確認待ち中のキャンセル）を
  起こさないこと。ここは "後で気づく" のではなく最初から設計に織り込む。

## 8. UI 変更（L0 と独立に先行導入も可）

- **送信ボタン ⇄ ■（停止）トグル**：処理中は送信ボタンが ■ に化ける（ChatGPT 型）。今の別枠キャンセル
  吹き出し（`cancelRow`）を廃し、入力エリアに集約。キャンセル＝キャンセルの意味は現状どおり。
- **追加条件の入力欄を本体に統合**：refine/overflow の専用 textarea を廃し、**自由入力は本体入力欄
  （`chat-input`）に一本化**。L0 ルーターが「新規／条件追加／確認回答／雑談」を振り分ける
  （今の `_hintResolve`/`_awaitingClarify`/`_feedbackResolve` の役目が L0 へ）。ボタン（提案・選択肢）は
  パネルに残す＝二枚看板。
- **`・・・` アニメーション**：L0 が喋る前に必ず `・・・`（作曲中の合図）→ 本文。**ユーザー入力中も同じ
  `・・・`** を出す（両者お揃いのモチーフ）。既存 `typing-dots` を流用。
- **開口一番（初回から）**：「**こんにちは、ジオネーターです！どこを探しますか？**」＝旧 L1-1 即応の後継。
- **入力保持**：キャンセル後に再入力不要。L0 がステートフルに文脈を持ち「さっきの『〜』続けますか？」で拾う
  （＝ L0 時代の自然形）。

## 9. デバッグモードの位置づけ（2 軸直交）

- **処理ビュー（表示量）**：詳細 ⇄ 会話 … *ユーザー向け*
- **デバッグ（調査ツール）**：ON/OFF … *運用者向け*（ステップ停止・schema JSON・グリッド円・TQ 内訳・地図デバッグ）

決定：**待ちナレーションを L0 に一元化した際に "降格" させたエンジン内部ラベル（場所特定/収集/評価）は、
デバッグモードで再露出する**。＝デバッグ＝「エンジン自身の声＋ステップ」、通常＝「L0 の声」。
デバッグは**詳細ビューの上に運用者オーバーレイを重ねたもので、L0 の会話も動かす**（本番と同じ挙動を観察）。
"純エンジンだけ見たい"（会話を隠す）サブトグルは必要になってから後付け。

## 10. 音声化（次段階）

会話が既にターン制テキストなので、音声化は**両端にアダプタを足す**のが基本：
- 入力 STT・出力 TTS。L0 出力はストリーム推奨（TTS 連携・会話らしさ）。
- **ナレーターは L0 一人**（決定 A のとおり。複数の声がかぶらない）。
- **バージイン**（喋りかぶせ → 中断/やり直し）＝既存 `_cancelled`/`_runGen`。
- **曖昧解消**：テキストはボタン維持、音声は**読み上げ選択肢＋発話で選ぶ**（選択肢集合はエンジンが決定的に
  計算、出し方だけモードで変える）。

## 12. 既存を壊さないための担保

- **L0 は "前置き" の追加**。エンジンの意思決定（駅特定・距離解釈・収集・スコア・tier）には触らない。
- エンジン側で触るのは**確認質問の "運び方"**（UICallbacks の中身を DOM 直行から L0 経由へ）だけ。
  **何を聞くかの判断は不変**、出口の配線だけ変える。
- 今回入れた各フローの「考え中」実装は**捨てない**：デバッグ表示＋ L0 が読むステータス源として活用。
- L1-1（`confirmInput`）は**コードを残す**（オフにするだけ）。

## 13. 確定した論点（旧・要確定ポイント）

1. **会話モードでのボタン → なし**。代わりに**候補・選択肢に番号を振り、ユーザーは番号を打つ**。
   番号 → index は JS が直接マップ（**完全決定的**・LLM 不要）、曖昧回答（「台東の方」等）のみ L0 が拾う。
2. **会話モードの地図ピン → タップ可能**。番号入力とタップの 2 経路（どちらも決定的）。
3. **確度ラベル → tier で確定**（§6）。別途のスコア差閾値は設けず、既に決定的な tier 分布を根拠にする。

## 14. 実装時に確認する技術前提（設計判断ではない）

- **LLM クライアントのストリーミング対応**：L0 の会話体感（`・・・` → 逐次表示）と将来の TTS 連携のため、
  `_callClaude` がストリームを返せるか要確認（未対応なら第 1 段の前提作業）。
- **L0 のモデル選択**：会話品質（Sonnet 系）か速度（Haiku）か。多ターンで履歴が伸びる＝トークン増も絡む。
  設定モーダルに L0 行を追加して切替可能にする想定（既定は要検討）。
