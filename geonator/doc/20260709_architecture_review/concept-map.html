Geonator · 機構棚卸し / MECE 監査

自然言語ロケーション探索 — 概念マップ

役割の3分割（proximity / target / condition）はMECE で健全。問題はその下の層——「場所の同定・曖昧解決」「親子（部位）」「フォールバック」が同じ問題に別々の機構を積み重ねて重複していること。これがオーナーが仕様を頭に保持できない主因。各軸を「1問題＝1系統」に統合するのが目標。

データモデル

QuerySchema（L1-2 の出力）

全機構はこの木の枝を解決する補助

自然文はまずこの1本の木に構造化される。以降の全軸（曖昧解決・部位・距離・フォールバック）は、結局この枝のどれかを確実に解決するための補助にすぎない。

QuerySchema
├─ proximity 基準（どこを起点に・常に1つ）
│ ├─ anchors[] 基準アンカー（通常1つ／「AとBの間」で2つ）
│ │ ├─ type ⟨station | poi | address | locality | intersection⟩
│ │ ├─ text
│ │ ├─ specificity ⟨specific | generic⟩
│ │ └─ subtype.exit 駅の出口（A4出口）→ transit系タイル
│ ├─ scope ⟨locality | place⟩ 県/市区でハード限定（岩手県の青山）
│ ├─ within ⟨level | 分=isochrone | m⟩ 明示された到達範囲
│ └─ bearing_filter ⟨north | south | east | west⟩
├─ target 探す物（常に1つ・答えとして表示）
│ ├─ text
│ ├─ query_intent ⟨category_mansion | category_apartment | category_building | specific⟩
│ ├─ queries[] QE（検索語）展開
│ └─ floors 階数（ハード条件）
├─ conditions[] 近接の制約（複数・重要順・OR評価）
│ ├─ type ⟨poi | road | water | rail | intersection | signal | transit_entrance | category_busstop⟩
│ ├─ text / queries[]
│ ├─ negate 否定（〜が無い）
│ ├─ direction
│ └─ distance
│ ├─ method ⟨radius | isochrone⟩
│ ├─ level ⟨same_building | adjacent | very_close | nearby | somewhat_nearby | far⟩
│ ├─ profile ⟨walking | cycling | driving⟩
│ └─ minutes / meters
├─ confirmation ユーザー向けの確認文
├─ unsupported_features[] 地図化できない特徴（築浅・壁が赤い等）
├─ result_area エリア表示モード（点でなく面）
└─ interpretations[] 解釈が割れる時のみ（最大3・→構造クラリファイ）
シーケンス

1クエリの処理フロー

JS主導：LLMは意味判断のみ

フェーズはコードの debug 表記に対応：⓪クエリ解釈 → ①一次検索(proximity解決) → ②二次検索 Step1(候補収集＋L2) → ③二次検索 Step2(距離評価・スコアリング) → ④結果・フィードバック。フロー制御・幾何・API裏取りは JS、LLM は解析・解釈・関連性の意味判断だけ。聞き返しは途中に差し込まれ選択後に続行。

背骨

役割（エンティティの3分割）

一貫 · 触らない

1クエリ＝proximity（基準）1つ・target（探す物）1つ・condition（制約）複数。ここは MECE で、全体の背骨。以降の軸はすべてこの3つの「解決」を助ける補助機構。

proximity

どこを基準に

  • anchors[] type: station/poi/address/locality/intersection
  • subtype.exit（駅出口）
  • scope（県/市区でハード限定）
  • within（到達範囲）
  • bearing_filter（方角）

target

何を探す

  • text（探す物）
  • query_intent mansion/apartment/building/specific
  • queries[]（QE展開）
  • floors（階数・ハード）

conditions[]

近接の制約

  • type poi/road/water/rail/…/category_busstop
  • distance level/method/minutes/meters
  • negate（否定）/ direction
  • queries[]（poiのみ）

軸 · 親子（部位）

「Xの部位/中のY」の表現

重複 3 + 穴 1

「親（建物）に従属する要素」という同じ関係が、4つの別概念に分散。データ有無で解決が変わるのが本質（駅出口＝在る / 通用口＝無い / 実在POI＝same_building）。

機構対象解決状態
subtype.exit + transit_entrance駅の出口（A4出口）transit系タイル（データ有）重複2表現
same_building実在POIが同ビル（大船駅のスタバ）建物ID一致（既存）実装済
container / childrenデータ無し部位（アミコの通用口）親建物にフォールバック未実装

統合先

「部位」1概念＋種別ディスパッチ解決：駅出口→transit / 実在POI→same_building / データ無し部位→親建物へ降格表示（＝真に新規で要るのはここだけ）。駅出口の既存路は壊さない。

軸 · 距離 / 範囲

「どこまで / どれだけ近い」

要ドキュメント化

収集extent（どこまで候補を集めるか）とマッチ距離（採点でどれだけ近いか）は別物。分離自体は正しいが、関係が明文化されておらず混同しやすい。

機構役割値状態
withinproximity到達（明示距離/時間）level / isochrone分 / m実装済
near-extent (type別)曖昧「近く」の収集半径POI400 / 駅600 / 地名800実装済
DISTANCE_TABLEconditionのlevel→半径/isoadjacent50…somewhat800実装済
condition margin収集bboxの拡張幅maxConditionRadiusM実装済

整理方針

2概念に明記：収集extent（広めOK・取りこぼし防止）／マッチ距離（採点・実距離で並ぶ）。「広め収集でも精度は採点が守る」を仕様書に1行で固定。

結論

統合後のMECEモデル ＆ 穴

目標＝この5軸だけで全部説明できる状態

  • エンティティ：役割(3) × 種別(POI/駅/エリア/部位/カテゴリ) × 親子(container)
  • 関係：距離・到達 / 包含(same-building) / 方角
  • 明確化＝2系統だけ：①場所同定（homonym/colloquial/broad/generic を1機構）②構造(interpretations)
  • フォールバック＝1ラダー：タイト条件 ▸ 親building ▸ 粗bbox ▸ 聞く
  • フィルタ：L2-1 カテゴリ / L2-2 名前（適用範囲を明記）

穴（優先度順）

  • 生きた概念仕様書が無い＝最大の穴。このマップを README 化してオーナーのグリップを維持
  • データ無し建物部位→親フォールバック（通用口）未実装
  • 岩手県 scope が L1 で確実に scope に入るか未検証（debug表示済）
  • condition側 L2-2 除外が debug 非可視（正当な候補が黙って落ちても見えない）
  • bbox閾値ズレ：広すぎゲート4km vs グリッド1500 ＝「2次検索は必ずTQ」が未保証

現状＝役割3はMECE / その下は 重複5(場所同定)・重複4(フォールバック)・重複3+穴1(部位)。 次の一手＝機能追加を止め、この軸へ写像する統合パス＋仕様書化。
