アイデア · 構想（未実装）

すべては located entity のマッチング

現状の proximity / target / condition という固定3型ツリーは、じつは 「座標を持つ entity が1種類あって、役割と関係が乗っているだけ」の特殊形。理想は、それを統一モデル＝空間マッチングエンジンとして捉え直すこと。速くはならないが、頭に収まり、いくらでも拡張できる。

出発点

なぜ作り直す価値があるか

target にファミマが来るか condition にファミマが来るか——違いは"役割"だけ。解決も QE も関連性判定も同じ。なのに型で分けているせいで、実装差と特殊ケースが増える。

現状

固定3型ツリー

  • proximity / target / condition が別々の型（役割が型に固着）
  • 多段チェーン（イオンモールの出口のそばのスタバの近くの公園）が表せない
  • 対称マッチ（Aにも Bにも近いファミマ）は役割反転でしか書けない
  • target/condition に本質でない実装差が溜まる
  • 部位フォールバック・同定・降格が個別機構で散る
理想

entity ＋ 関係 ＋ 役割

  • entity は1種類。proximity/target/condition は役割ラベル
  • チェーンは関係グラフで自然に表現
  • 対称マッチは「どれを出力にするか」を変えるだけ
  • 収集・QE・関連性は完全対称（1経路）
  • 同定・降格・部位はこの1モデル上で統一

統一モデル

Entity ・ Relationship ・ Role

Entity（1種類）

座標を持つ場所。resolve種別＝名前POI / カテゴリ / 駅 / エリア / 建物の部位。text・queries・container（親建物）を持つ。

Relationship

entity 間の空間関係。距離/到達（円・isochrone）・包含（same-building / 中）・方角。

Role（役割ラベル）

output＝ランク＆表示（1）／ reference＝距離で測る側（N・共有可）／ region＝探索の器。

出力候補  ×  参照集合  の空間マッチング  ＋  関連性採点

これで埋まる穴

統一すると一気に解ける

✓
多段チェーン
「イオンモールXの出口のそばのスタバの近くの公園」＝ entity を関係で繋ぐグラフ。特殊処理不要。
✓
対称マッチ
「マンションにも駅にも近いファミマ」＝ ファミマを output にするだけ。役割反転が自然。
✓
target/condition の実装差を廃止
収集・QE・L2-2 関連性を1経路に統一。分岐とバグ源が減る。
✓
建物部位フォールバック
通用口＝ entity の resolve種別「データ無し部位」。不在なら container（親building）へ降格＝共通ラダー。
✓
明確化=2系統に収束
① entity 同定（場所が曖昧）／ ② 構造（役割・関係の割り当てが割れる）。今の5+機構がこの2つに。
✓
フォールバック=1ラダー
output 不在 → タイトな reference → 親 container → 粗 → 聞く。降格の一本化。

現実的な移行

理想へどう寄せるか（忖度なし）

段階移行（デモは止めない）

  1. 概念を先に採用：頭のモデルを「entity＋関係＋役割」に更新（この一枚）。現状マップの用語もこれに寄せる。
  2. 内部の対称化から：収集/QE/L2-2 を target/condition で共通化（既にほぼ対称＝低リスク）。
  3. L1 出力を「役割ラベル付き entity ＋ 関係」に：固定3枠 → entities[] ＋ relations[] ＋ output指定。ただしよくある形は固定3枠のショートカットを残す（L1の負担と失敗面を抑える）。
  4. 関係グラフ評価器：現在の「候補×条件の最近傍採点」を、任意の関係グラフを解く形に一般化。

正直なトレードオフ：一般グラフは固定3枠より失敗面が広いし、速くはならない（得るのは MECE性と拡張性）。だからデモは固定ツリーで安全に走らせつつ、内側を段階的に寄せる。「どれを output にするか」という唯一の非対称だけは、どのモデルでも残る。

これは理想（未実装の構想）。現状の実装は別ページ「Geonator 概念マップ」を参照。／ 唯一の本質的非対称＝出力の指定。それ以外は対称化できる。
